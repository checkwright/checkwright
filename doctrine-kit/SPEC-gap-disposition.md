# SPEC amendment: gap-disposition

## What changes

`DOCTRINE.md` gains one new methodology-maintenance rule, appended after
Spec-over-precedent as **rule 10 — Gap disposition**: when a session surfaces
a gap it will not fix this session (a coverage hole, a design defect, an
escaping class), it *costs the remedy* — works out how the gap could be
closed, not just names it — and lands that analysis where the next session
finds it: a new queue entry, or an enrichment of the standing deferred entry
that owns the gap. A bare flag-and-skip is the defect. This is the
knowledge-friction loop one altitude up: that loop captures re-derived *facts*
with no owner; this rule captures *design and coverage gaps*. It fills the
space between two existing rules: Spec-over-precedent captures a doc gap met
during precedent consultation; Enforcement-first governs a fix being *landed*;
neither governs a gap being *deferred*.

**Numbering ruling.** The rule is appended at the end of the methodology block
(becomes rule 10), so the engineering-craft block renumbers 10–19 → 11–20.
This is the one renumber event of the iteration — the sibling doctrine
amendments (audit-cadence, artifact-verification, identifier-deliteralization)
extend existing rules precisely so no second renumber lands. Before the
renumber, every by-number craft-rule citation migrates to a name-based one:
the known instances are the history-rewrite advisory's three "DOCTRINE.md
rule 15" citations — one in `guard-kit/lib/guard.sh`, two in the advisory's
own entry in guard-kit/SPEC.md §The generic ruleset (the checklist reference
and the collision note, which must be reworded rather than renumbered) → cite
the rule by its name, "Re-verify volatile state before a git history rewrite"
— and the build session sweeps the tree for any other `DOCTRINE.md rule
1[0-9]` citation before renumbering (Rename is a full-surface sweep). The
sweep matches the doc-qualified form only: guard-kit numbers its own ruleset
1–16 independently, and its internal by-number citations ("rules 12, 13",
"rule 14") are that ruleset's, never DOCTRINE.md's — an unqualified
`rule 1[0-9]` sweep would mis-touch them.

**Enforcement ruling.** Stated manual duty, no gate — "did the session cost
the remedy" is not machine-decidable (the Enforcement-first false-positive
carve-out; Spec-over-precedent's "judgment with a capture mechanism, not a
gate" is the model). The rule's *Enforced by* line names the capture
mechanisms it rides: the queue's deferred section (design gaps) and the
knowledge-friction log (fact gaps).

## Producers and consumers

- **Producer:** any session, at the moment a gap is surfaced-and-deferred; the
  rule text is the trigger, loaded via the doctrine link and the CLAUDE.md
  digest bullet.
- **Consumer:** the scope stage — an enriched deferred entry arrives at
  scope's triage carrying its costed remedy instead of a bare mention; the
  queue entry grammar is unchanged (no new tag, no new field).
- **Digest lockstep:** the CLAUDE.md digest is generated by
  `doctrine-kit/bin/install-doctrine.sh` (its `block()` emits one bullet per
  methodology rule), so the same unit adds the rule's one-line bullet to
  `block()` and re-runs the installer; `check-doctrine-registration` holds the
  per-rule lockstep and is the gate that reddens a digest that lags.

## Definition of Done

      reachable producer and a named consumer; every new field has a named
      reader at a named transition.

- [ ] **Amendment deleted** — this file removed on merge; none remain for the
      component.
- [ ] **Removals propagated** — grepped every spec and source for by-number
      craft-rule citations; nothing dangles after the renumber.

      as debt tasks.
